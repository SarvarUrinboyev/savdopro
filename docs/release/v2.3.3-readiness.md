## 3. Production gated steps (do in order; STOP on any failure)
1. Merge `release/v2.3.3-ui-import-ready` → `main`; tag **v2.3.3**; push tag → GitHub Actions deploys.
2. Health-check the live backend; confirm the version/tag is live.
3. **Backup** the production DB (see the import runbook).
4. **Preview** import (read-only) — confirm shop `Asosiy do'kon`, 6 598 rows, 8 negative-corrected, 0 dup barcodes.
5. **Confirm the target shop** is the intended production shop (id + name).
6. **Import** (~13 min at 540 req/min), then re-run for idempotency.
7. **Verify** (count, code/name search, duplicate-name group, 30 samples, negatives = 0, other shops untouched).

Full commands: [`docs/import/warehouse-excel-production-runbook.md`](../import/warehouse-excel-production-runbook.md).

## 4. Rollback
- **App:** redeploy the previous tag (v2.3.2) via the same GitHub Actions flow.
- **DB:** restore the §3.3 backup (`pg_restore` for Postgres, or swap the `.mv.db` for H2). The importer never deletes, so the only forward-written data lives in the target shop.

## 6. Remaining risks
- **P1 — accounting canonical currency:** ledger is single-scale (sales/COGS raw, expenses converted). See `docs/pilot/ACCOUNTING_RULES.md`. Not changed by this release.
- **P2 — fractional stock:** 142 Excel rows have fractional quantity, rounded to integer on import (model is `int`). Weight-based items lose the fraction.
- **P2 — unit default "dona":** the Excel "Ед.из" column is a numeric MXIK package code, not a human unit, and has no Product field — products import with unit `dona`.
- **P2 — backup/restore rehearsal:** rehearse the restore once before the production import.
- **P3 — Z/X report & monitoring:** confirm daily-close report and basic uptime/error monitoring before wider rollout.
