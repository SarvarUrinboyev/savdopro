# Release v2.3.4 — Production Hardening + Import Safety

## Why a new tag (not a re-tag of v2.3.3)

`v2.3.3` is already deployed in production as commit `8726df51` (== `origin/main`).
The import-safety commit `3274f72` ("lock warehouse import to explicit account
and shop ids") was added to the release branch **after** the PR #4 merge, so it
is **NOT** in `v2.3.3` (`git merge-base --is-ancestor 3274f72 8726df51` → false).
Re-using `v2.3.3` would be dishonest and would confuse the deployed history.
**Use `v2.3.4`** to carry the import-safety commit + the production hardening.

## Chosen strategy — Option B (branch → PR → tag)

Recommended over a direct merge because it gets a review gate on production-facing
security config.

1. Branch `fix/prod-profile-security-hardening` **off the release branch**
   (`3274f72`) so it already contains the import-safety commit. ✅ done.
2. Add the hardening commits (this change).
3. Open a PR into `main`. Review focuses on: prod profile activation, `ddl-auto`,
   CSP compatibility, CORS origins, systemd/secret handling.
4. Merge to `main`.
5. Tag `v2.3.4` on the merge commit.

> ⚠️ **Tagging triggers deploy.** `.github/workflows/deploy.yml` runs on
> `push: tags: ["v*"]`. Do **not** push `v2.3.4` until the founder has approved
> the deploy window, taken a DB backup, and is ready to run the
> [production-hardening-runbook](../ops/production-hardening-runbook.md) (the
> systemd/env/secret steps are **manual**; CI only swaps the jars).

## What ships in v2.3.4

**Repo (auto-deployed with the jars):**
- Prod profile: `ddl-auto=validate` (env-overridable), `forward-headers-strategy=framework`,
  demo-seed hard-off, explicit actuator include.
- Backend `SecurityConfig`: HSTS, CSP (prod-only, SPA-verified), Referrer-Policy,
  Permissions-Policy (X-Frame-Options DENY / X-Content-Type-Options nosniff kept).
- License `SecurityConfig`: HSTS, Referrer-Policy, Permissions-Policy;
  env-gated `forward-headers-strategy`, loopback-bind note.
- Swagger/OpenAPI already prod-disabled — now *effective* because the prod
  profile is actually activated (via the runbook).

**Manual on the server (runbook — NOT in the jars):**
- `--spring.profiles.active=prod` via `SPRING_PROFILES_ACTIVE` in a 0600 env file.
- Dedicated `savdopro` / `savdopro-license` service users; owned runtime dirs.
- Secrets moved out of the 644 `application-local.properties` into `/etc/savdopro/*.env`.
- systemd hardening drop-ins (see `ops/systemd/`).

## Order of operations (must not reorder)

1. Merge PR → tag `v2.3.4` (founder-approved window) → CI builds + deploys jars.
2. Apply the hardening runbook on the server (users, env files, drop-ins,
   remove the local override, `--profiles.active=prod`), license first then backend.
3. Verify (profile=prod, api-docs is SPA-HTML not JSON, health 200, 401, CORS
   fail-closed, 6 headers, ports on 127.0.0.1).
4. **Backup the DB**, then run the Excel import **preview** (only when explicitly
   requested), from the `v2.3.4` code / this branch — never from `main`/`v2.3.3`.
5. License H2 → Postgres migration is a **later, separate** change.

## Must NOT happen
- No `v2.3.3` re-tag. No tag push before approval. No import from `main`/`v2.3.3`.
- No production restart without founder go-ahead. No DB migration on prod by hand
  (Flyway runs on service start). No secrets committed.
